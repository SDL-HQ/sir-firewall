# D12 — S1.1 Structured mode design (Wave 1, C-Round)

Date: 2026-04-16  
Status: design only (no implementation)

## 1) Design objective

Define one optional structured-input path that improves determinism for a narrowly bounded high-stakes request class, without turning SIR into a workflow engine.

This note is intentionally narrow:

- one request path
- one request class
- one schema shape
- deterministic reject behavior
- no new public truth surface

## 2) Selected structured request path (single path)

### Path name (design label)

`account_recovery_challenge_request_v1`

### Bounded request class (single class)

Inbound account-recovery requests that ask for reset/credential takeover-sensitive actions.

This class is chosen because:

- it is explicitly high-stakes (identity takeover and fraud risk)
- it already aligns with existing SIR evaluation scope (request-level text gating, account-recovery fraud domain material)
- it benefits from deterministic field constraints before text rule evaluation

No second class is included in this round.

## 3) Mode semantics (optional, not universal)

Structured mode is an **optional** alternate input envelope for the single class above.

- If not used, existing text-first request path remains unchanged.
- If used, request must match this schema exactly.
- Structured mode does not add orchestration, state transitions, or policy-execution logic.

## 4) Canonical example schema (design-level)

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "$id": "sir.account_recovery_challenge_request.v1",
  "title": "SIR Structured Request: Account Recovery Challenge v1",
  "type": "object",
  "additionalProperties": false,
  "required": [
    "schema_version",
    "request_class",
    "action",
    "channel",
    "request_text"
  ],
  "properties": {
    "schema_version": {
      "type": "string",
      "const": "v1"
    },
    "request_class": {
      "type": "string",
      "const": "account_recovery_challenge"
    },
    "action": {
      "type": "string",
      "enum": [
        "password_reset",
        "mfa_reset",
        "email_change",
        "phone_change"
      ]
    },
    "channel": {
      "type": "string",
      "enum": [
        "chat",
        "email",
        "support_ticket"
      ]
    },
    "request_text": {
      "type": "string",
      "minLength": 1,
      "maxLength": 4000
    },
    "declared_auth_state": {
      "type": "string",
      "enum": [
        "verified",
        "unverified",
        "unknown"
      ]
    },
    "case_ref": {
      "type": "string",
      "pattern": "^[A-Za-z0-9._:-]{1,64}$"
    }
  }
}
```

### Minimum allowed fields

Required minimum:

- `schema_version`
- `request_class`
- `action`
- `channel`
- `request_text`

Optional fields are intentionally few:

- `declared_auth_state`
- `case_ref`

## 5) Deterministic validation and fail behavior (design)

Validation is fail-closed and deterministic.

### Deterministic fail conditions

Reject when any of the following occur:

1. Not valid JSON object.
2. Missing any required field.
3. Unknown top-level field present (`additionalProperties: false`).
4. Field type mismatch (for example non-string where string required).
5. Enum violation (`action`, `channel`, `declared_auth_state`).
6. `schema_version` or `request_class` const mismatch.
7. `request_text` length outside bounds.
8. `case_ref` fails strict pattern.
9. Duplicate keys in input object (if raw parser exposes duplicates) — reject as malformed.
10. Mixed-mode stuffing attempt: nested objects/arrays or encoded payload smuggling in non-text fields — reject because fields are scalar-only by schema.

### Deterministic handling rule

- Structured-validation failure returns a single non-passing deterministic outcome before inference.
- No coercion, no auto-repair, no best-effort parse, no fallback to free-form mode for the same payload.

## 6) Request-level boundary (why this is not workflow logic)

This design remains pre-inference request-level because it does only:

- fixed-shape envelope validation
- bounded field extraction
- deterministic pass/fail gating eligibility

It does not do:

- case lifecycle progression
- identity verification state machines
- multi-step challenge orchestration
- downstream action execution

## 7) Evidence/provenance coherence (no new truth surfaces)

Structured mode must remain coherent with existing SIR evidence surfaces.

Design constraints:

- keep existing artefact surfaces as source of truth (`run_summary`, ITGL, certificates, run archives)
- do not add new public pages, dashboards, or parallel truth endpoints
- represent structured-mode usage via existing evidence channels only (for example existing run/certificate metadata fields where available in current contract)
- if validation fails, outcome is still captured through existing non-passing run evidence semantics

No provenance model refactor is proposed in this round.

## 8) Not for (explicit out-of-scope expansions)

Structured mode in this round is **not for**:

- generic enterprise JSON intake
- arbitrary custom schemas per customer/team
- workflow or policy engines
- conditional branching/action graphs
- tool/function-call payload governance
- multimodal request envelopes
- conversational/session memory governance
- post-inference moderation orchestration
- analytics/scoring/telemetry additions
- parser architecture redesign
- CLI dispatch expansion

## 9) Smallest future implementation surface (validator boundary only)

Smallest next-step implementation boundary:

- a single structured-request validator at request ingress that:
  - checks strict schema for `account_recovery_challenge_request_v1`
  - returns deterministic validation result (accept/reject + fixed reason code)
  - emits normalized scalar values needed by existing request-level gate path

This is intentionally limited to validator boundary definition only.
No parser architecture, dispatch model, workflow behavior, or broader interface design is included.

## 10) Bounded rationale

Why this is the narrow best fit now:

- High-stakes and concrete: account recovery fraud is materially sensitive.
- Determinism gain: strict field envelope reduces ambiguity before text rules.
- Product-boundary safe: remains request-level, pre-inference, text-centered.
- Audit-safe: failure outcomes and usage can remain inside existing evidence surfaces.
- Minimal blast radius: one schema, one class, one validator boundary.
